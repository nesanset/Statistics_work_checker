# Statistics Work Checker

Веб-приложение для проверки контрольных работ по статистике.

Проект разработан в рамках курсовой работы по дисциплине «Теория и технологии программирования». Основной пользователь системы - преподаватель, который проверяет работы студентов, выставляет оценки за задания и формирует отчеты по результатам проверки.

## Назначение проекта

При ручной проверке контрольных работ преподавателю приходится отдельно искать студента, определять его вариант, открывать файл варианта и сверять задания с максимальными баллами. При большом количестве студентов такой процесс становится долгим и неудобным.

Statistics Work Checker автоматизирует этот процесс:

- импортирует список групп и студентов из xlsx-файла;
- импортирует варианты заданий из папки с xlsx-файлами;
- сопоставляет студентов с их вариантами;
- показывает задания выбранного студента;
- позволяет выставлять, изменять и удалять оценки;
- поддерживает шаблоны комментариев;
- автоматически считает суммарный балл;
- определяет статус студента относительно проходного балла;
- формирует сводный отчет по группе.

## Основной функционал

В приложении реализованы:

- регистрация и вход преподавателя;
- хранение паролей в виде BCrypt-хеша;
- создание контрольной работы;
- изменение проходного балла;
- импорт xlsx-файлов со студентами и вариантами;
- просмотр групп выбранной контрольной работы;
- просмотр студентов выбранной группы;
- просмотр заданий варианта конкретного студента;
- выставление оценки за задание;
- удаление оценки;
- отметка «Нет работы» с автоматическим выставлением нулевых оценок;
- шаблоны комментариев: «Без комментария», «Задание не выполнено», «Выполнено корректно»;
- автоматический расчет суммы баллов;
- автоматическая маркировка относительно проходного балла;
- формирование отчета по группе.

## Архитектура

Приложение построено по многослойной архитектуре:

- `web` - REST-контроллеры и DTO-ответы для клиентской части;
- `service` - бизнес-логика приложения;
- `domain` - доменные модели и бизнес-правила;
- `persistence` - JPA-сущности и репозитории;
- `resources/static` - клиентская веб-страница.

Поток обработки запроса:

1. Пользователь выполняет действие на веб-странице.
2. JavaScript отправляет HTTP-запрос на сервер.
3. Контроллер принимает запрос.
4. Сервис выполняет бизнес-логику.
5. При необходимости сервис обращается к репозиториям.
6. Репозитории работают с PostgreSQL через Spring Data JPA.
7. Данные преобразуются в ответ и возвращаются на веб-страницу.

## Паттерны проектирования

В проекте применены несколько паттернов.

### Facade

`ControlWorkImportFacade` скрывает детали импорта xlsx-файлов. Внешний код запускает импорт через один метод, а фасад внутри организует чтение списка студентов, чтение файлов вариантов и передачу разобранных данных на сохранение.

### Chain of Responsibility

Проверка оценки реализована через цепочку валидаторов:

- оценка должна быть указана;
- оценка не должна быть отрицательной;
- оценка не должна превышать максимальный балл задания;
- оценка должна быть кратна 0.25;
- при нулевой оценке нельзя выбрать комментарий «Выполнено корректно».

Каждое правило вынесено в отдельный валидатор, поэтому цепочку можно расширять без переписывания всей логики проверки.

### Repository

Доступ к данным реализован через Spring Data JPA Repository. Сервисы не пишут SQL-запросы напрямую, а используют репозитории для сохранения и чтения сущностей.

### Mapper

Мапперы отделяют модели разных слоев:

- Entity - данные для хранения в БД;
- Domain - бизнес-модели;
- Response - объекты ответа для REST API.

## База данных

В качестве СУБД используется PostgreSQL.

Схема базы данных описана в файле:

```text
src/main/resources/init.sql
```

При запуске приложения Spring Boot выполняет этот скрипт и создает необходимые таблицы, ограничения и индексы. Hibernate работает в режиме `validate`, то есть проверяет соответствие JPA-сущностей готовой схеме базы данных.

Основные таблицы:

- `app_users` - пользователи системы;
- `control_works` - контрольные работы;
- `student_groups` - группы студентов;
- `students` - студенты;
- `variants` - варианты заданий;
- `assignments` - задания;
- `grades` - оценки.

Основные связи:

- один пользователь может создать несколько контрольных работ;
- одна контрольная работа содержит группы и варианты;
- одна группа содержит студентов;
- один вариант содержит задания;
- студент имеет назначенный вариант;
- оценка связывает студента и задание.

## Настройка базы данных

Перед запуском нужно создать базу данных PostgreSQL.

Пример имени базы:

```text
statistics_work_checker
```

Параметры подключения задаются в `src/main/resources/application.yml` или через переменные окружения:

```text
DB_URL
DB_USERNAME
DB_PASSWORD
```

По умолчанию приложение использует:

```text
DB_URL=jdbc:postgresql://localhost:5432/statistics_work_checker
DB_USERNAME=User
DB_PASSWORD=
```

Если на компьютере другой пользователь PostgreSQL, нужно изменить переменные окружения или настройки в `application.yml`.

```bash

```


## Демонстрационные данные

В проекте есть готовые xlsx-файлы для проверки импорта.

Первый набор:

```text
sample-data/students.xlsx
sample-data/variants
```

Второй набор:

```text
sample-data/students_extra.xlsx
```

Файл студентов содержит столбцы:


Файлы вариантов содержат задания на последнем листе. Ожидаемые столбцы:

```text
Текст задания

```

## Демонстрационный сценарий

Для проверки основного сценария можно выполнить следующие действия:

1. Запустить приложение.
2. Открыть `http://localhost:8080`.
3. Зарегистрировать нового пользователя.
4. Создать контрольную работу.
5. Указать проходной балл.

   - файл студентов: `sample-data/students.xlsx`;
   - папка вариантов: `sample-data/variants`.
7. Выбрать группу.
8. Выбрать студента.
9. Выставить оценки за задания.
10. Проверить автоматический расчет суммы баллов.
11. Выбрать шаблон комментария.
12. Проверить действие «Нет работы».
13. Сформировать отчет по группе.

## Особенности реализации

- Проверка оценок выполняется на сервере, поэтому некорректные значения не сохраняются в базе данных.
- Пароли пользователей хранятся в виде BCrypt-хеша.
- Оценка не может превышать максимальный балл задания.
- Оценка должна быть кратна 0.25.
- Для пары «студент - задание» может быть сохранена только одна оценка.
- При отметке «Нет работы» студенту автоматически выставляются нулевые оценки по всем заданиям.
- Отчет формируется на основе данных из БД и воспроизводимо показывает текущие результаты проверки.
